# Sentence Library Generation

The tailored sentences shown in Practice (Sentence mode) come from a bundled
static corpus exported from the public Supabase table, `public.sentences`, keyed
by `(word_key, type)`. Corpus chunks are loaded on demand and runtime-cached by
the service worker; deterministic local templates cover custom words and
offline misses when a chunk has not been cached yet. This doc is the operational
runbook for filling that table and exporting the app's sentence corpus.

The pipeline makes **no LLM API calls**. Sentences are generated by **Codex**
(run manually on your machine via the ChatGPT Pro plan); the scripts only emit
work and validate/import/export the results. At runtime, the app checks the
bundled corpus first, then the Supabase table, then the deterministic local
template generator for custom words or any missing row.

## How it fits together

1. `npm run sentences:batches` writes JSONL batch files of pending
   `(word, conjugation)` pairs, plus a `PROMPT.md`.
2. **Codex** reads each batch and writes a matching output JSONL of sentences.
3. `npm run sentences:import <output.jsonl>` validates each sentence against the
   conjugation engine, derives per-token furigana with **kuromoji**, and upserts
   to Supabase. Rejected lines go to `<output>.rejects.jsonl`.
4. `npm run sentences:check-corpus` compares the checked-in offline corpus to
   Supabase without writing files.
5. `npm run sentences:export-corpus` reads Supabase and writes chunked JSON to
   `public/data/sentences`, one file per type under `by-type/`, plus a manifest.

The conjugated form is recomputed from the engine and must appear verbatim in the
sentence, so a wrong tense/form is rejected rather than stored.

## One-time setup

Install deps and set the Supabase service credentials (the **`service_role`**
key from Supabase → Project Settings → API — not the anon key):

```bash
npm install
export SUPABASE_URL="https://<your-project-ref>.supabase.co"
export SUPABASE_SERVICE_ROLE_KEY="<service_role key>"
```

Optionally regenerate the lexicon so offline fallbacks get real JMdict
transitivity (needs network), and commit the result:

```bash
npm run vocab:build
git add public/data/verb-lexicon.json
git commit -m "Regenerate verb lexicon with transitivity"
```

## Environment variables

| Var | Default | Meaning |
| --- | --- | --- |
| `SENTENCE_JLPT` | all levels | Comma list of JLPT levels to include, e.g. `N5,N4`. |
| `SENTENCE_TYPES` | **all 126 forms** | Comma list of type ids, `all`, or `default` (the 28 learner-default forms). |
| `SENTENCE_BATCH_SIZE` | `200` | Pairs per batch file. |
| `SENTENCE_LIMIT` | `0` (no cap) | Cap total pairs emitted (for smoke tests). |
| `SENTENCE_OUT_DIR` | `tmp/sentence-batches` | Where batches/outputs are written (gitignored). |
| `SUPABASE_URL`, `SUPABASE_SERVICE_ROLE_KEY` | — | Required to import; when set, `sentences:batches` skips pairs already in the DB (resumable). |
| `SENTENCE_DRY_RUN` | unset | `1` validates without upserting. |
| `SENTENCE_NO_KUROMOJI` | unset | `1` uses model-provided segments instead of kuromoji. |
| `SENTENCE_TEMPLATE_CAP` | `100` | Max times one sentence template may be reused across a single import invocation (`0` disables). |
| `SENTENCE_CORPUS_OUT_DIR` | `public/data/sentences` | Where `sentences:export-corpus` writes the bundled corpus. The path must resolve to a nested directory named `sentences` because the exporter clears it before writing. |

## Quality gates (anti-stub)

The importer rejects, to `*.rejects.jsonl`, any line that isn't a genuine
tailored sentence:

- `en` that contains Japanese, has no letters, is boilerplate
  (e.g. "A short practice sentence using … form."), or names the grammar form.
- The conjugated `expected_surface` not appearing verbatim in `ja`.
- Furigana segments that don't tile the sentence, or non-kana readings.
- A sentence template reused more than `SENTENCE_TEMPLATE_CAP` times.

The template cap is enforced across everything in **one** `sentences:import`
invocation, so pass all outputs at once (e.g. `out-*.jsonl`) for it to apply
globally rather than per file.

`sentences:batches` now defaults to **every form for every word at every JLPT
level** — a plain run covers everything the engine can produce. Narrow it with
`SENTENCE_JLPT` / `SENTENCE_TYPES` when you want a smaller tranche.

**Defective verbs are pruned at the engine.** A few verbs don't inflect across
the full matrix, so the conjugator reports the missing forms as non-existent
(`''`) and the pipeline never asks for — or accepts — a sentence for them: ある
(在る・有る) and the godan 要る are limited to their core everyday forms (no
potential, passive, causative, progressive, desiderative, keigo, volitional,
command, or request), and 生まれる drops only its nonexistent passive and
causative-passive. This is the same `''` contract used elsewhere (e.g.
short-causative-passive on ichidan verbs) and applies everywhere
`practiceTypesForItem` runs, including core practice — not just the corpus. See
`STATIVE_DEFECTIVE_SUPPRESSED` / `SPONTANEOUS_INTRANSITIVE_SUPPRESSED` in
`src/utils/conjugator.js`.

## Run everything (instructions for Codex)

Paste the following to Codex. With `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`
set, it will work through the entire lexicon, resuming where it left off.

> **Task: fill the tailored sentence library for every word and form.**
>
> The repo has a generation pipeline; you (Codex) are the generator. The scripts
> make no LLM calls. `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY` are set.
>
> 1. `git checkout main && git pull && npm install`
> 2. Emit batches for everything: `npm run sentences:batches`
>    (every word × every applicable form, skipping pairs already in the DB).
>    This writes `tmp/sentence-batches/PROMPT.md` and `batch-0001.jsonl`, …
> 3. Read `tmp/sentence-batches/PROMPT.md`. Then process **every** `batch-*.jsonl`
>    file in order. For each batch, write a matching `out-*.jsonl` where each line
>    is `{ "word_key", "type", "ja", "en" }`:
>    - `ja` is one short, natural, beginner-appropriate sentence that uses the
>      word in the requested form and contains `expected_surface` **verbatim,
>      exactly once**.
>    - `en` is a real English translation of that sentence — NOT a description of
>      the task, no Japanese, and never names the grammar form.
>    - Write a different, context-appropriate sentence per word; do not reuse one
>      template for everything (the importer caps template reuse).
>    - Use the `transitive` hint and `type_label` to phrase the form correctly.
>    - Do not include readings; furigana is derived automatically. JSONL only.
> 4. Validate: `SENTENCE_DRY_RUN=1 node scripts/import-sentence-library.js tmp/sentence-batches/out-XXXX.jsonl`
>    Fix any lines listed in `out-XXXX.rejects.jsonl` (usually the sentence
>    didn't contain `expected_surface`) and re-validate until clean.
> 5. Import: `node scripts/import-sentence-library.js tmp/sentence-batches/out-XXXX.jsonl`
>    (idempotent upsert by `(word_key, type)`).
> 6. Continue to the next batch file until all are imported. If interrupted,
>    re-run `npm run sentences:batches` — it regenerates only the pairs still
>    missing from the DB.
>
> Notes: `tmp/` is gitignored — do not commit batch/output files. The only thing
> worth committing is a regenerated `public/data/verb-lexicon.json` from
> `npm run vocab:build`.

## Repairing legacy English glosses

Older generated output files may have structurally valid Japanese sentences but
generic English like "A short practice sentence using ...". To repair those
outputs before import, rewrite only the `en` field into a separate temp folder:

```bash
npm run sentences:english -- tmp/sentence-batches tmp/sentence-natural-english
```

Then validate/import the corrected `out-*.jsonl` files from
`tmp/sentence-natural-english`. If you are reseeding this legacy corpus exactly
as-is, set `SENTENCE_TEMPLATE_CAP=0` during import because the legacy Japanese
frames intentionally reuse a small fixed template set; do not use that override
for new generation runs.

## Checking progress

```sql
select count(*) from public.sentences;
select type, count(*) from public.sentences group by type order by 2 desc;
```

Imported sentences appear in Practice → Sentence mode automatically for the
covered words. Run `npm run sentences:check-corpus` to prove whether the
checked-in static corpus already matches Supabase exactly. If it reports drift,
run `npm run sentences:export-corpus` so those sentences ship as static app
data; the exporter fails if any current built-in word/form pair is missing from
Supabase. Full lexicon × all forms is ~150k–270k sentences, so expect a long
run; Supabase value lands continuously as batches import, while bundled static
value lands with the next app deploy.

## Exporting the offline corpus

```bash
npm run sentences:check-corpus
npm run sentences:export-corpus
```

Both commands require `SUPABASE_URL` and `SUPABASE_SERVICE_ROLE_KEY`, read only
from `public.sentences`, and filter out stale rows that no longer match the
current lexicon/form matrix. The check command compares exact generated file
contents against `public/data/sentences` and performs no writes or deletes. The
export command writes:

- `public/data/sentences/manifest.json`
- `public/data/sentences/by-type/<type>.json`

Each type file has this compact shape:

```json
{ "schema": 1, "type": "plain-past", "rows": [["word_key", "ja_template", "en", []]] }
```

Do not hand-edit the exported files. Regenerate them from the validated
Supabase table.

The service worker does not precache the manifest or every `by-type/*.json`
chunk during app install because the full corpus is large. Runtime checks the
small manifest with a network-first strategy, builds a version fingerprint from
that manifest, and adds it as a query string when loading each type chunk. Type
chunks still use a cache-first strategy once loaded, but a regenerated corpus
gets fresh chunk URLs so corrected sentences reach returning learners. If the
manifest or a chunk is unavailable offline before it has been cached, Sentence
mode falls back to deterministic local templates.
